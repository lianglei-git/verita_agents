### 1. 默认展示逻辑

#### 1.1 打开方式
- 双击聊天窗口中的图片缩略图，打开独立图片查看器窗口。
- 支持键盘左右箭头切换上/下一张图片（如果聊天记录中有多张）。

#### 1.2 窗口与画布适配
查看器窗口分为两种状态：**嵌入窗口模式**（默认）和**全屏模式**（双击窗口或点击全屏按钮）。

**核心规则：以较长边为准**

- **图片尺寸 > 窗口尺寸（超比例视图）**：
  - 图片自动缩放，**保持长宽比**，完整显示在窗口可视区域内。
  - 此时图片四周可能留有“黑边”或“灰边”（取决于背景色），以确保图片不变形。
  - 光标默认显示为放大镜图标（`zoom-in`），表示可放大。

- **图片尺寸 <= 窗口尺寸（小比例视图）**：
  - 图片以**原始尺寸（100%）** 展示，不进行拉伸放大。
  - 如果图片小于窗口，图片居中显示，边缘留白（背景色填充）。
  - 光标默认显示为手型或正常箭头（视具体版本而定），通常无法直接拖动（因为没有溢出）。

### 2. 鼠标滚轮缩放逻辑

滚轮是PC端查看图片最核心的缩放入口。

- **缩放中心**：以**鼠标光标所在位置**为中心点进行缩放。
- **缩放方向**：
  - 向前滚动（向上滚）：**放大**。
  - 向后滚动（向下滚）：**缩小**。
- **缩放步长**：每次滚动，图片大小变化为当前尺寸的 **10% - 15%**（需平滑过渡）。
- **最小比例限制**：缩小至图片**宽度或高度有一边填满窗口**为止？严格来说，微信逻辑是**最小不能低于“适应窗口”模式**。即无论怎么缩小，不能缩得比“初始打开时的完整显示”还要小。
  - *逻辑解释*：当用户缩小到“完整显示”状态时，继续执行缩小操作无效，或者滚动无法再缩小。
- **最大比例限制**：通常限制为原始图片像素的 **1000% 或 物理极限（如最大6400px）**，防止像素无限拉伸导致性能问题。

### 3. 鼠标拖动缩放逻辑

这里指通过**按住鼠标左键并拖动鼠标**（通常配合快捷键）或**触控板手势**。

- **PC微信标准逻辑**：
  - **未放大时（图片完整显示）**：**不支持拖动**。拖动图片不会移动图片，而是触发切换到下一张图片（如果鼠标在边缘）或无反应。
  - **放大后（图片超出窗口可视区域）**：光标变为“手型”（`grab`/`grabbing`），按住左键可以**拖拽移动**画布。
- **拖拽方向**：全方向移动（上下左右）。
- **拖拽惯性**：PC微信通常没有惯性滑动（即刻停止），保持跟手性即可。

### 4. 最大边界逻辑

当图片被放大并拖拽时，必须限制拖拽范围，防止图片完全移出视野导致用户“找不到图片”。

- **边界限制规则**：**允许溢出，但不允许露出空白**。
  - 图片边缘可以移动到窗口边缘，但**不允许图片边缘越过窗口边缘后露出背景色**（除非图片尺寸小于窗口）。
  - **具体规则**：
    - 当图片宽度 > 窗口宽度时：横向拖拽，图片的左侧边缘最远只能移动到窗口左边缘（此时右侧仍在窗口外），反之亦然。不能把整张图拖到只剩空白。
    - 当图片宽度 < 窗口宽度时：图片自动居中对齐，拖拽失效（或复位）。
  - **回弹效果**：如果用户试图拖拽超过边界，应感受到阻力（边缘阻尼），松手后自动回弹至合法边界（通常不做强制回弹，而是硬边界限制）。

### 5. 按钮与辅助交互

除了鼠标手势，UI控件也是需求的一部分：

- **缩放按钮**：
  - **“+”按钮**：点击一次，放大一步（以图片中心为基准）。达到最大比例时置灰。
  - **“-”按钮**：点击一次，缩小一步。达到“适应窗口”比例时置灰（或继续点无效）。
- **“1:1”按钮**（通常隐藏或替代方案）：点击显示图片实际像素（100%）。
- **“适应窗口”按钮**（旋转箭头或方框图标）：点击后，图片恢复默认打开状态（完整显示在窗口内），重置缩放和位置。
- **旋转**：支持右键或工具栏旋转（左转/右转）。旋转后，重新计算“适应窗口”的适配逻辑。

### 6. 多图切换时的状态重置逻辑

这是容易被忽略但非常重要的交互细节。

- **场景**：用户查看图片A，放大了200%并拖拽到了右下角，此时点击“下一张”切换到图片B。
- **需求**：
  - **重置状态**：切换到新图片时，**缩放比例和拖拽位置必须重置**，恢复为该图片的默认展示状态（适应窗口）。
  - **不应**：带着上一张图的缩放比例和偏移量查看下一张图，否则会导致下一张图直接处于“丢失”或过度放大的状态。
- **记忆逻辑**：如果用户返回上一张图，是否需要记忆之前的缩放状态？
  - PC微信通常**不记忆**。每次切换都是重新加载默认视图。这是为了降低用户认知负担，确保每次切换看到的都是完整的图。

### 7. 窗口尺寸改变逻辑

当用户拖拽改变查看器窗口的大小时：

- **实时重绘**：
  - 如果当前图片处于**“适应窗口”模式**（默认视图）：图片跟随窗口等比缩放，始终完整显示。
  - 如果当前图片处于**“放大”模式**（比例固定）：窗口变大时，可视区域变大，图片尺寸保持不变；如果窗口变得比图片大，图片仍然保持放大状态（不自动缩小回适应模式），但拖拽边界会重新计算。

### 8. 键盘快捷键

- **ESC**：关闭查看器。
- **方向键 ← / →**：切换上一张/下一张。
- **Ctrl + 滚轮**：虽然在图片查看器中滚轮就是缩放，但在PC微信中，标准滚轮直接缩放，无需Ctrl辅助。
- **Space（空格）**：部分版本支持，点击空格切换“适应窗口”模式。

### 9. 特殊边界情况

- **超大图（如 10000x8000）**：
  - 打开时默认适应窗口。
  - 缩放时限制最大缩放比例，防止内存溢出（通常限制为原始尺寸的100%或略高，视物理内存而定，微信可能会提示“图片太大，无法继续放大”）。
  - 拖拽流畅度需保证，建议使用虚拟渲染或缓存。
- **长图（如手机截图，高度极高）**：
  - 默认适应窗口宽度，高度超出屏幕。
  - 此时默认状态下即可**垂直滚动**（通过鼠标滚轮滚动内容）？
  - PC微信逻辑：默认适应宽度，如果图片高度超过窗口，**用户可以使用鼠标滚轮垂直滚动图片**（此时滚轮行为需定义）。
    - *关键分歧点*：当图片高度大于窗口（且宽度适配）时，滚轮是应该**缩放**还是**滚动**？
    - **主流PC微信逻辑**：如果图片处于默认视图且溢出窗口，滚轮负责**滚动查看溢出部分**；如果需要缩放，需点击“+”或使用Ctrl+滚轮（部分版本优化不同，需明确）。需求文档建议：**以光标位置判断，如果没有溢出，滚轮缩放；如果有溢出且光标在图片上，优先滚动查看溢出区域，配合Ctrl缩放。**

- **GIF动图**：
  - 默认展示为播放状态。
  - 缩放时保持动态播放。
  - 右下角显示“GIF”标识，且通常不显示进度条，只有“大小”信息。

- **二维码识别**：
  - 右键菜单中应有“识别图中二维码”功能（如果有二维码）。
  - 这是微信特有的强需求。

### 10. 性能与内存

- **懒加载**：多图切换时，只加载当前图片和前后各一张，释放非可见区域的图片内存。
- **高清图**：对于微信发送的高清原图，查看时先显示缩略图模糊占位，再加载清晰大图（渐进式加载）。

| 模块 | 需求点 | 描述 |
| :--- | :--- | :--- |
| **初始展示** | 超比例 | 缩放至完整显示，留黑边，光标放大镜 |
| | 小比例 | 显示原始尺寸，居中，光标默认 |
| **滚轮缩放** | 中心点 | 以鼠标位置为中心 |
| | 步长 | 10%-15% 平滑变化 |
| | 范围 | 最小：适应窗口；最大：100% 或 物理上限 |
| **拖拽** | 触发条件 | 仅当图片放大超出窗口时可用 |
| | 边界 | 硬边界，不允许露出空白背景 |
| **多图切换** | 状态重置 | 缩放归位，位置归位，恢复默认视图 |
| **窗口缩放** | 响应 | 适应窗口模式跟随缩放；放大模式不跟随 |
| **特殊格式** | 长图 | 默认适配宽度，滚轮滚动内容 |
| | GIF | 动态播放，支持缩放 |
